## Context

当前项目已经具备用户登录、SkillGateway、SSH Skill 以及 SkillHub 入口的基础能力，但服务器连接信息仍然依赖调用侧直接传入，缺少可持久化、可维护、按用户隔离的服务器台账。用户希望把服务器 `ip`、`username`、`password` 记录在 Java 数据库中，并在执行 SSH Skill 时直接根据当前用户台账中的目标服务器解析登录信息，而不是在请求中重复传递敏感凭据。

该变更同时涉及后端数据模型、接口权限、SSH 执行链路和前端管理界面，属于跨模块改造。现有用户体系已经支持登录和用户切换，因此设计需要复用当前用户身份作为台账归属依据，并保证不同用户之间的数据完全隔离。

## Goals / Non-Goals

**Goals:**
- 在 `backend/skill-gateway` 中建立服务器台账持久化模型，并与用户建立一对多关联
- 提供当前用户维度的服务器台账 CRUD 接口，支持查看列表和维护记录
- 让 SSH Skill 在执行时通过“当前用户 + 目标 IP”解析登录信息，并复用既有 SSH 执行链路
- 在前端增加服务器台账入口与维护界面，支持列表查看、新增、编辑、删除
- 明确密码字段的读写边界，避免在列表接口中无必要暴露敏感值

**Non-Goals:**
- 不引入外部 Secret Manager、KMS 或额外基础设施
- 不支持跨用户共享服务器台账
- 不在本次变更中实现批量导入导出、连通性探测或密码轮转
- 不扩展到 SSH 之外的其他 Skill 凭据管理模型

## Decisions

### 1. 使用独立的服务器台账实体并按用户做唯一约束
- 新增 `ServerLedger`（命名可在实现阶段微调）实体，核心字段包含 `id`、`userId`、`ip`、`username`、`password`、`createdAt`、`updatedAt`
- 数据库存储在现有 Java 数据库中，不引入新的存储系统
- 对 `(user_id, ip)` 建立唯一约束，避免同一用户重复维护相同 IP 的凭据；不同用户可以登记相同 IP

这样可以直接满足“每个用户拥有独立服务器台账”的需求，并为 SSH Skill 的解析提供稳定的数据来源。

备选方案：
- 直接把服务器字段挂到现有 `Skill` 实体下。放弃原因是服务器台账本身是独立业务对象，不应依附 Skill 定义存在。
- 允许同一用户对同一 IP 维护多条记录。放弃原因是会让 SSH Skill 的解析出现歧义。

### 2. 后端接口始终基于当前登录用户推导归属，不接受外部传入 userId
- 新增面向当前用户的接口，例如 `GET /api/server-ledgers`、`POST /api/server-ledgers`、`PUT /api/server-ledgers/{id}`、`DELETE /api/server-ledgers/{id}`
- controller/service 层从当前认证上下文获取用户身份，查询和写入时自动附带用户约束
- 任何对不属于当前用户的记录访问，都返回未找到或无权限错误，不暴露其他用户数据是否存在

这样能把“用户隔离”落实到接口契约层，避免前端或 Agent 伪造 `userId`。

备选方案：
- 前端显式传 `userId`。放弃原因是客户端不可信，且会放大越权风险。

### 3. SSH Skill 继续以目标主机标识发起调用，由服务端按“当前用户 + IP”解析凭据
- 保持 SSH Skill 面向 Agent 的输入尽量简化，执行请求至少包含 `ip`（或兼容已有 `host` 字段）和 `command`
- SkillGateway 在执行前先校验命令安全性，再使用当前用户与目标 IP 查询服务器台账，取出 `username`、`password`
- 查询成功后复用现有 SSH 执行服务；查询失败时直接返回明确错误，不尝试默认连接

这样可以让 Agent 继续围绕“对哪台机器执行什么命令”思考，而不必处理密码等敏感信息。

备选方案：
- 改为使用 `serverId` 查找台账。优点是主键稳定；放弃原因是当前需求明确强调“根据对应 IP 获取登录信息”，且对 Agent 来说直接按 IP 指定目标更自然。

### 4. 列表接口不返回明文密码，维护接口通过显式写入覆盖密码
- `GET /api/server-ledgers` 返回 `id`、`ip`、`username` 以及必要的展示字段，但不返回明文 `password`
- 新增和编辑接口接受 `password` 输入；编辑时若未传新密码，可保持原值不变，若传入则覆盖
- 前端列表页展示服务器基础信息；密码仅在新增/编辑表单内输入，不在列表中直接回显

这样在满足“台账维护”的同时，降低密码在 UI 和网络响应中的暴露面。

备选方案：
- 列表接口返回完整密码。放弃原因是没有必要，且明显增加泄露风险。

### 5. 前端采用独立台账入口和维护弹窗，而不是复用 SkillHub
- 在主界面增加“服务器台账”按钮，作为与 SkillHub 并列的管理入口
- 进入后展示当前用户服务器列表，并提供新增、编辑、删除操作
- 新增/编辑使用表单弹窗或抽屉，字段至少包括 `ip`、`username`、`password`

服务器台账属于资源管理，而 SkillHub 更偏能力发现。两者职责不同，分离入口更清晰。

## Risks / Trade-offs

- [风险] 密码保存在业务数据库中，敏感性较高
  - 缓解：列表接口不返回明文密码；后续可在实现阶段增加加密存储或脱敏日志约束
- [风险] 现有登录态在 SkillGateway 中的用户解析方式可能不统一
  - 缓解：在实现前先统一当前用户获取方式，优先复用已有认证上下文
- [风险] 使用 `ip` 作为 SSH Skill 解析键会要求用户维护准确且唯一的 IP
  - 缓解：对同一用户增加唯一约束并在新增/编辑时做重复校验
- [风险] 删除或修改台账记录后，依赖该 IP 的 Agent 调用可能立即失败
  - 缓解：在错误响应中明确说明“当前用户不存在该服务器台账或凭据不可用”

## Migration Plan

1. 在数据库中新增服务器台账表及与用户的关联字段
2. 发布后端 CRUD 接口与 SSH 台账解析逻辑
3. 前端发布服务器台账管理入口和维护界面
4. 将 SSH Skill 调用改为依赖台账解析，停止从上层透传登录凭据

如需回滚，可以先禁用前端入口并回退后端解析逻辑；数据库表保留不会影响既有用户登录与聊天主流程。

## Open Questions

- 当前 SSH Skill 请求体是否已经固定为 `host` 字段，还是可以平滑切换到 `ip` 字段并兼容旧参数？
- 密码字段在实现阶段是否需要直接明文持久化，还是可以同时引入应用层加密而不改变外部需求？
