## Context

当前 `skills` 表为平台级单表，无 `userId` 或可见性字段；SkillGateway 的列表与 CRUD 未按用户隔离。产品需要区分「所有人可用的公共 Skill」与「仅创建者可见可改的私人 Skill」，并要求所有经生成工具写入的 Skill 绑定当前登录用户。Agent Core 通过 `GET /api/skills` 拉取扩展 Skill，必须与同一套可见性规则一致，否则会泄露私人工具或误注册他人配置。

## Goals / Non-Goals

**Goals:**

- 在持久化层为每条 Skill 记录建立 **可见性**（`PUBLIC` / `PRIVATE`）与 **创建者**（`createdBy` 或与现有用户主键一致的字符串 ID）。
- **Built-in Skills** 一律视为 **公共**，作者/归属固定为 **`public`**（与真实用户新建 Skill 的 `createdBy` 区分）；Hub 与 Agent 对内置能力的语义与扩展库私人 Skill 分离。
- **新建** 用户侧数据库 Skill（表单 / API / 生成）在未指定可见性时 **默认 `PRIVATE`**；用户显式选公共则保存为 `PUBLIC`。
- 列表接口返回 **当前用户可见** 的子集：全部 `PUBLIC` + `PRIVATE` 且 `createdBy` 等于当前用户。
- 更新/删除：仅创建者可修改或删除 `PRIVATE` Skill；`PUBLIC` Skill 的写权限默认与当前实现一致（通常为任意已登录用户或后续可收紧为管理员，本变更若未引入角色则保持「已登录可写公共」需在实现中与产品确认）。
- 前端在新增/编辑表单提供「公共 / 私人」选择，**新建时默认选中私人**；生成类路径写入时自动设置创建者为当前用户，未传可见性时默认 **PRIVATE**。

**Non-Goals:**

- 不实现复杂的组织/团队共享、Skill 转交、或细粒度 RBAC。
- 不引入除「创建者 + 公共/私人」以外的第三态（如「链接分享」）。
- 不在本变更中要求前端为历史数据批量修改可见性（仅迁移默认值即可）。

## Decisions

### 1. 数据模型：字段命名与约束

- **决策**：在 `Skill` 实体增加 `visibility`（枚举：`PUBLIC` | `PRIVATE`）与 `createdBy`（可空仅用于迁移期；新创建必须非空）。
- **理由**：与提案用语「公共/私人」一致；`createdBy` 与现有 `User` 主键类型对齐（与 `ServerLedger.userId` 相同约定）。
- **备选**：仅用 `ownerUserId` 不设枚举——放弃，因公共 Skill 无单一「所有者」语义，需显式可见性。

### 2. 私人 Skill 的读权限

- **决策**：`GET` 列表与详情均不返回「他人私人」Skill；Agent 与前端使用同一列表 API 规则。
- **理由**：避免 Hub 与 Agent 行为不一致；详情若按 ID 直链访问，无权限时返回 404 或等价空结果，不泄露存在性（与现有台账越权语义一致）。

### 3. 公共 Skill 的写权限

- **决策**：在尚未引入管理员模型时，**允许任意已登录用户更新/删除公共 Skill**（与当前全局表行为接近），或 **仅创建者可改公共 Skill**——需在实现前与产品二选一。
- **建议默认**：若当前线上已依赖「任何人可改公共配置」，为减少 **BREAKING** 行为，迁移后公共 Skill 仍允许已登录用户修改；若产品强约束「仅创建者」，则在本变更 tasks 中明确并一次性收紧。
- **本设计占位**：spec 中写「对已认证用户可管理的范围按 `skill-visibility` 规则」，实现任务中锁定具体策略。

### 4. 历史数据迁移

- **决策**：现有行：`visibility=PUBLIC`，`createdBy=NULL` 或系统占位 ID（若需非空则使用固定 `system` / 首管理员，以迁移脚本为准）。
- **理由**：保持历史 Skill 对所有人可见；创建者未知时不赋予私人语义。

### 5. Agent 请求身份

- **决策**：Agent Core 调用 SkillGateway 时必须携带与前端一致的 **用户身份**（已有 JWT / Header 则复用）；若当前 Agent 调用为匿名，需在本变更中打通「运行会话用户」到 Gateway，否则无法实现「本人私人 Skill」。
- **理由**：否则列表无法过滤私人 Skill。

### 6. 生成工具默认可见性

- **决策**：生成工具持久化时设置 `createdBy=当前用户`，`visibility=PRIVATE`（除非用户在流程中显式选公共）。
- **理由**：贴合「生成的是我的实验 Skill」；与提案「通过 skill 生成工具生成的 skill 属于当前登录用户」一致。

### 7. Built-in Skills 为公共且作者为 public

- **决策**：所有 Built-in Skills（Hub 内置区块、Agent 内置工具）语义上为 **公共**；若存在持久化行，则 `visibility=PUBLIC` 且 `createdBy=public`（或与用户表中的系统用户 `public` 对齐）。
- **理由**：与产品定案一致——内置能力不属于某一真实用户，不得标为私人或随用户切换而「变成私人」。

### 8. 新建数据库 Skill 默认私人（全路径）

- **决策**：除 Built-in 种子/系统写入外，**任意**新建数据库 Skill（手工、REST、生成器）在未显式指定可见性时 **默认 `PRIVATE`**；与 §6 生成工具策略一致，避免遗漏某一入口默认为公共。
- **理由**：减少个人实验默认暴露给全平台；用户需主动选「公共」才共享。

## Risks / Trade-offs

- **[Risk] 公共 Skill 被任意用户误改** → **Mitigation**：文档说明；后续可加「仅创建者可改」或管理员角色。
- **[Risk] Agent 未带用户身份导致私人 Skill 永不加载** → **Mitigation**：在 tasks 中明确串联认证；集成测试覆盖「登录用户 + 私人 Skill」路径。
- **[Risk] 迁移后唯一性仍全局按 `name`** → **Mitigation**：私人 Skill 是否允许与用户范围内重名需在实现中定义（建议：`name` 仍全局唯一，避免 Agent 工具名冲突；若需按用户重名，改为复合唯一约束，工作量大，本阶段非目标）。

## Migration Plan

1. 数据库迁移：添加列与索引（如 `(visibility, createdBy)` 查询优化视需要）。
2. 后端部署：新代码读新列；旧行默认值 `PUBLIC`。
3. 前端与 Agent：发布后用户可逐步为 Skill 标私人；生成路径自动写入创建者与默认私人。
4. **Rollback**：回滚应用版本；若需回滚 DDL，保留列可空避免丢数据，或备份后删列（谨慎）。

## Open Questions

- 公共 Skill 的 **编辑/删除** 是否仅限创建者（与当前「全局可改」的取舍）。
- `name` 全局唯一 vs 每用户唯一：若维持全局唯一，用户 A 的私人 Skill 名称与用户 B 冲突时的提示文案与策略。

### 落地结论（实现阶段）

- **公共 Skill 写权限**：维持为 **任意已登录用户**（请求带 `X-User-Id` 且通过 `X-Agent-Token` 的写操作）可更新/删除 `PUBLIC` Skill，与历史全局表行为一致。
- **`name` 唯一性**：维持 **全局唯一**；重复名称创建时后端返回「Skill with name … already exists」；生成器在列表不可见他人私人同名 Skill 时，若 POST 撞库仍由网关返回错误。
