## Context

当前 `agent-core` 已经通过 `LoggerService` 把 LLM 输入、输出、中间思考与工具结果写入 `logs/llm.log`，但写法是追加纯文本块，既没有稳定的结构，也没有和单次会话/当前对话建立前端可消费的关联。聊天前端目前只接收 assistant 文本与工具状态相关的 SSE 事件，没有“日志查看按钮”、日志面板，也没有日志状态管理。

这次需求是一个跨模块改造：`agent-core` 需要把现有 LLM 日志升级为结构化事件；任务相关接口/事件流需要把这些日志增量暴露给前端；聊天界面需要提供一个基于当前对话的日志查看入口，并在会话进行中实时刷新日志内容。用户特别关心“送给大模型的参数”和“模型返回的内容”必须被清楚区分，因此日志模型不能继续停留在自由文本。

## Goals / Non-Goals

**Goals:**
- 为单次对话/任务建立结构化 LLM 日志模型，明确区分请求参数与模型返回内容。
- 支持日志随当前对话实时更新，而不是只写本地文件供线下排查。
- 在聊天界面提供“日志查看”按钮，并可打开查看当前对话中的 LLM 请求/响应日志。
- 保留日志落盘能力，同时让落盘内容与前端展示使用同一结构化模型。
- 让日志按会话或任务维度隔离，避免不同对话内容混杂。

**Non-Goals:**
- 不把所有后端内部调试信息都暴露给用户；首版聚焦 LLM 请求与响应相关信息。
- 不做完整的日志检索平台或跨历史会话全局搜索。
- 不在本次变更中引入第三方日志/Tracing 基础设施。
- 不默认展示完整敏感字段脱敏策略以外的原始密钥或凭据。

## Decisions

1. **以结构化日志事件替代纯文本块作为主模型**
   - **决定**：定义统一的 `LlmLogEntry` 结构，至少包含 `sessionId/taskId`、时间戳、方向（request/response）、阶段、摘要、payload；`llm.log` 由此结构序列化写入，前端展示也消费同一模型。
   - **理由**：同一套数据模型既能支持前端实时展示，也能保留文件排查价值，避免双份格式长期漂移。
   - **替代方案**：继续写纯文本，并在前端侧二次解析。这样实现快，但无法稳定区分字段，也不利于实时流式更新。

2. **日志按当前对话/任务维度关联**
   - **决定**：每条结构化 LLM 日志都必须带当前任务或会话标识，前端“日志查看按钮”只查看当前对话关联的日志。
   - **理由**：用户要的是“当前对话过程中”的模型调用记录，而不是全局日志文件片段。
   - **替代方案**：只按用户维度聚合日志。这样实现简单，但多轮并发对话会混杂。

3. **通过实时事件流暴露日志更新，而不是轮询整个日志文件**
   - **决定**：任务进行过程中，通过现有任务事件流或等价实时通道增量发送结构化 LLM 日志事件；前端据此实时更新日志面板。
   - **理由**：该需求强调“根据对话内容实时更新”，事件流比轮询文件更低延迟，也更容易绑定当前任务上下文。
   - **替代方案**：前端定时拉取日志接口。这样简单但有时延，也增加无效读取。

4. **日志查看入口放在聊天对话上下文中，而不是单独全局页面**
   - **决定**：在当前聊天界面增加“日志查看”按钮，并打开一个日志抽屉/弹窗，展示与当前 assistant 会话相关的日志条目。
   - **理由**：日志要服务于当前对话调试和解释性，放在会话上下文中最容易被理解。
   - **替代方案**：做独立日志页面。适合全局运维，但切换成本高，也不符合“当前对话”视角。

5. **请求侧与响应侧分栏/分组展示**
   - **决定**：日志查看器中明确区分“发送给大模型的参数”和“模型返回内容”，请求侧至少展示模型名、消息列表、工具绑定/调用相关参数，响应侧至少展示文本内容、tool calls 或结构化返回。
   - **理由**：这是用户提出的核心识别需求，必须在 UI 与数据模型层同时体现。
   - **替代方案**：仅按时间线平铺，不强调方向。会降低可读性，不满足需求。

6. **日志内容做必要脱敏与裁剪，但保持可调试性**
   - **决定**：结构化日志不得包含 API key 等敏感凭据；对于体积过大的 messages/响应，可展示摘要并允许查看展开后的结构化内容。
   - **理由**：日志既要可查看，也不能把敏感配置直接暴露在前端。
   - **替代方案**：完整原样透传。调试最方便，但风险过高。

## Risks / Trade-offs

- **[Risk] LLM 请求/响应 payload 过大，导致 SSE 噪音和前端渲染压力上升** → **Mitigation**: 事件流传输结构化摘要与必要字段，完整内容按条目展开或按需读取。**
- **[Risk] 不同模型/SDK 返回结构不一致，难以统一展示** → **Mitigation**: 先抽象成稳定的 request/response 核心字段，再把差异放入扩展 payload。**
- **[Risk] 日志里混入敏感信息或用户隐私** → **Mitigation**: 在写入结构化日志前做字段白名单/脱敏处理，禁止直接记录密钥。**
- **[Risk] 当前对话与日志条目关联不稳，导致查看器展示错会话数据** → **Mitigation**: 强制每条日志带任务/会话标识，并由前端按当前会话过滤。**
- **[Risk] 同时保留文件日志和实时事件，代码路径可能重复** → **Mitigation**: 统一从同一个 `LlmLogEntry` 生产函数分发到文件与实时通道。**

## Migration Plan

1. 先定义结构化 `LlmLogEntry` 模型，以及请求/响应的字段边界与脱敏规则。
2. 改造 `agent-core` 的 LLM 日志写点，使其生成结构化日志对象，并按当前任务/会话打标。
3. 通过任务事件流或配套接口把日志增量暴露给前端客户端。
4. 在前端聊天界面增加日志查看按钮、日志状态容器与日志展示面板。
5. 最后补充联调与验证，确认日志随对话实时更新，且请求参数与响应内容分区清晰。

回滚时可先关闭前端日志入口，并让后端退回纯文本落盘；如果结构化字段仍保留在文件中，也不影响旧逻辑读取。

## Open Questions

- 实时日志更新应完全复用现有任务 SSE，还是单独增加“日志流”接口以避免和 assistant 文本事件混杂？
