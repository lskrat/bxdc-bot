## Why

在低版本浏览器或不同操作系统上，**原生 Unicode emoji 的字体与字形不一致**，用户可能看不到注册时选的「头像」emoji，或显示为方块/与预期不符。需要在前端用**统一渲染源**（如 Twemoji 等）展示头像与会话中的 emoji，减少环境差异。

同时，用户登录后应能**自行修改昵称与头像**，但 **用户 ID（6 位数字）为账户主键**，必须保持**不可变更**，避免破坏会话、记忆与权限绑定。

## What Changes

- **Emoji 展示统一**：在聊天、侧栏、设置等展示用户/助手头像及必要 emoji 的位置，采用前端引入的**统一图片或字体策略**（具体技术见 design）。**支持范围**为 **Chrome / Chromium 内核 86 及以上**；在此范围内头像展示须一致、可辨，并与常见桌面/移动系统协调。
- **资料编辑**：新增「编辑资料」能力：用户可更新 **昵称**、**头像**（emoji 选择或与现有一致的 AI 生成流程）；**禁止**修改 **用户 ID**（界面不展示可编辑 ID 或提交时忽略/拒绝）。**用户编辑自己的资料时不得要求输入管理员密码**（仅需已登录用户身份，与现有一致鉴权方式）。
- **后端**：若尚无「仅更新昵称」接口，则扩展 `User` 更新 API（与现有 `PUT /api/user/{id}/avatar` 对齐风格）；校验昵称长度与头像格式与注册一致。
- **非目标**：不改变用户 ID 的生成规则；不强制将所有正文里的 emoji 都转为图片（可按产品只覆盖头像与关键 UI）。

## Capabilities

### New Capabilities

- `emoji-display-unification`：定义跨浏览器/跨系统 emoji 头像与相关 UI 的统一展示策略与可访问性底线。

### Modified Capabilities

- `user-profile`：增加「登录后编辑昵称与头像、ID 不可变」的要求。
- `chat-ui`（可选补充）：消息列表/头像区对当前用户头像的展示与统一 emoji 策略一致。

## Impact

- **frontend**：新依赖（如 `@twemoji/api` 或 `twemoji` CDN 封装）、封装 `Avatar`/emoji 渲染组件；设置或侧栏增加「编辑资料」表单；**构建/测试对齐 Chromium 86+**。
- **skill-gateway**：`UserController` / `UserService` 扩展更新昵称（及合并资料更新）；数据库字段已有 `nickname`/`avatar`，通常无需迁移；**资料更新接口不得绑定管理员密码**。
- **向后兼容**：已有用户仅更新展示层与编辑入口，**无 BREAKING API** 若采用新增字段或兼容 PUT。
