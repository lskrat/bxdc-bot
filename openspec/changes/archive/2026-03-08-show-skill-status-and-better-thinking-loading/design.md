## Context

当前前端聊天页基于 Vue 3 和 `@tdesign-vue-next/chat` 构建，`useChat` 仅维护消息文本、错误状态和 `isThinking` 布尔值。它能够消费 SSE 文本流并拼接 assistant 回复，但没有单独的数据结构记录 Skill 调用过程，因此界面无法展示“正在调用哪个 Skill、是否完成”。

后端方面，`skill-gateway` 负责对外暴露 `/api/tasks/{id}/events`，并把 `agent-core` 的流式事件直接转发给前端。`agent-core` 当前主要把 LangGraph chunk 原样写入 SSE，前端只提取其中的文本内容，没有对工具调用事件做统一建模。因此，本次改动是一个跨前后端的流式协议增强，而不仅是局部样式优化。

## Goals / Non-Goals

**Goals:**
- 为每条 assistant 回复维护独立的 Skill 调用状态列表，支持按顺序展示多个 Skill。
- 让前端能够区分 Skill 的开始、进行中、完成和失败状态，并在成功时显示小对勾。
- 将当前三个点 loading 升级为更明确的“思考中 + emoji + 动态效果”体验，且与流式回复生命周期一致。
- 尽量复用现有 SSE 通道，避免引入额外轮询或第二条状态请求链路。

**Non-Goals:**
- 不引入复杂的 Agent 执行时间线、步骤树或可折叠调试面板。
- 不改变任务创建接口的基础调用方式，仍保留 `POST /api/tasks` + `GET /api/tasks/{id}/events` 模式。
- 不在本次变更中重做整套聊天 UI 组件，仅对消息内容区下方的辅助信息和 loading 展示做增强。

## Decisions

### 1. 为 assistant 消息增加结构化执行元数据
- **决策**：在前端消息模型中新增与文本内容分离的执行状态字段，例如 Skill 调用列表、整体 thinking 状态与完成态。
- **理由**：当前 `Message` 只有 `content`，如果把 Skill 状态直接混入正文，会污染 Markdown 内容，也无法单独做轻量样式和动态更新。
- **备选方案**：
  - 将 Skill 调用状态直接拼进 assistant 文本中：实现简单，但会破坏正文可读性。
  - 单独维护全局状态面板：无法准确对应到某一条 assistant 回复，不符合“展示在 AI 返回对话下面”的需求。

### 2. 使用现有 SSE 链路传递结构化 Skill 事件
- **决策**：继续复用当前任务 SSE 通道，由 `agent-core` 归一化工具/Skill 调用事件，再由 `skill-gateway` 原样透传给前端。
- **理由**：用户需要实时看到调用状态，SSE 已存在且适合单向流式更新；只需要在现有事件模型中补充结构化字段即可。
- **备选方案**：
  - 前端根据文本推断 Skill 调用：不可靠，且受模型输出内容影响。
  - 为 Skill 状态新增独立 API：会增加状态同步复杂度和竞态风险。

### 3. 前端按“顺序列表 + 增量更新”渲染多个 Skill
- **决策**：每次收到 Skill 事件时，前端按 Skill 标识匹配已有条目；不存在则追加，已存在则更新状态。
- **理由**：一次回复可能触发多个 Skill，也可能同一 Skill 经历 started/running/completed 多次更新，顺序列表最符合用户感知。
- **备选方案**：
  - 仅显示当前正在执行的一个 Skill：会丢失历史轨迹。
  - 每个状态变化都追加新行：容易产生重复记录，噪声较大。

### 4. loading 改为可定制的 thinking 文案与动效
- **决策**：保留现有聊天组件的流式能力，但自定义展示为“思考中 + emoji + 微动效”，避免继续使用默认的三个点。
- **理由**：当前默认 loading 识别度低，且与用户期望的 Agent 思考反馈不匹配。轻量动画可以在不大改布局的前提下提升感知质量。
- **备选方案**：
  - 仅替换为纯文本“思考中”：可读性提升有限，视觉表现仍然偏弱。
  - 引入新的动画库：实现成本更高，也会增加依赖与样式维护成本。

## Risks / Trade-offs

- **Skill 事件结构与 LangGraph 实际输出不一致** → 先在 `agent-core` 做归一化层，避免前端直接依赖底层 chunk 形态。
- **同一条消息的文本流与 Skill 状态流更新顺序交错** → 以前端消息 ID 作为聚合单位，分别更新正文和执行元数据。
- **默认聊天组件对 loading UI 的定制能力有限** → 如插槽能力不足，则在组件外补充自定义 thinking 区域，而不是强依赖内建三个点样式。
- **失败态定义不清晰** → 设计上至少保留成功/失败两种终态，失败文案与视觉可先保持轻量，后续再扩展。

## Migration Plan

1. 在 `agent-core` 中识别并归一化 Skill/工具调用事件，输出稳定的 SSE 数据结构。
2. 保持 `skill-gateway` 事件转发链路不变，仅确保结构化事件可被原样传递到前端。
3. 扩展 `frontend/src/composables/useChat.ts` 的消息模型与事件解析逻辑，支持 Skill 状态列表。
4. 更新消息组件，在 assistant 回复下方渲染小字体的 Skill 调用状态，并增加成功对勾。
5. 替换 loading 展示，验证文本流、Skill 状态流和完成事件在正常/异常情况下均能正确收敛。

## Open Questions

- 是否需要在界面上区分“正在调用 Skill”与“调用后等待模型继续生成”的两个不同 thinking 阶段？
- 若未来除了 Skill 还会展示其他工具调用，当前状态文案是否统一使用“工具/技能”术语，还是按来源分别展示？
