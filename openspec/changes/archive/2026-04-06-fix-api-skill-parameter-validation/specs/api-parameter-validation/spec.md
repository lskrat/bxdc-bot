## ADDED Requirements

### Requirement: Strict API Parameter Validation
The system MUST validate the parameters generated by the LLM against the API skill's defined `parameterContract` (JSON Schema) before executing the API request.

#### Scenario: Valid parameters
- **WHEN** the LLM generates parameters that fully comply with the `parameterContract`
- **THEN** the system proceeds to execute the API request with the provided parameters

#### Scenario: Invalid parameters (missing required field)
- **WHEN** the LLM generates parameters but omits a field marked as required in the `parameterContract`
- **THEN** the system MUST NOT execute the API request
- **AND** the system MUST return a structured error message to the LLM indicating which required field is missing

#### Scenario: Invalid parameters (incorrect type)
- **WHEN** the LLM generates parameters where a field's value does not match the type specified in the `parameterContract` (e.g., string instead of integer)
- **THEN** the system MUST NOT execute the API request
- **AND** the system MUST return a structured error message to the LLM indicating the type mismatch

#### Scenario: Invalid parameters (enum violation)
- **WHEN** the LLM generates parameters where a field's value is not one of the allowed values in the `parameterContract` enum
- **THEN** the system MUST NOT execute the API request
- **AND** the system MUST return a structured error message to the LLM indicating the allowed enum values

### Requirement: Error Feedback Loop
When parameter validation fails, the system MUST return the validation errors as the tool execution result, allowing the LLM to observe the failure and attempt to correct its output.

#### Scenario: LLM corrects parameters after validation error
- **WHEN** the system returns a validation error to the LLM
- **THEN** the LLM receives the error message
- **AND** the LLM can generate a new tool call with corrected parameters based on the error feedback
