## Context

当前 API Skill 同时把三类信息塞进一套配置里：一类是给 Agent 做路由判断的简要用途描述，一类是给人维护和给模型理解的详细接口说明，另一类是运行时真正执行请求所需的参数格式约束。这种混合结构在前端表单、后端动态注册和 `skill_generator` 生成链路里都已经产生耦合：

- 前端结构化表单难以表达“简述”和“详细说明”的不同用途。
- Agent 在初始注册工具时拿到的信息要么过少，要么过多，无法做到按需披露。
- API Skill 运行前缺少统一的参数契约校验，模型输出一旦格式偏差就可能直接打到外部接口。
- `skill_generator` 只能生成较粗粒度配置，无法稳定产出可执行且可校验的 API Skill。

这次变更同时涉及 `frontend` 的 Skill 编辑体验、`backend/agent-core` 的 Skill discovery 与 runtime 执行，以及内置 API Skill 生成功能，属于跨模块重构。

## Goals / Non-Goals

**Goals:**

- 为 API Skill 建立更清晰的配置分层：简要 `description`、详细接口说明、参数格式契约。
- 在 Agent 路由阶段隐藏详细接口说明，在实际调用阶段再注入。
- 在发出外部请求前对模型输出做严格参数校验，并返回结构化错误。
- 让 `skill_generator` 生成的新 API Skill 默认符合上述结构，减少后续人工修正。

**Non-Goals:**

- 不重构 SSH Skill、OPENCLAW Skill 或其他非 API Skill 的配置协议。
- 不引入新的外部校验服务或新的数据库表。
- 不要求本次变更覆盖响应体的严格 schema 校验；本次重点是请求参数契约与接口说明治理。

## Decisions

### 决策 1：将 API Skill 拆成“路由摘要 + 接口说明 + 参数契约”

- 方案：保留现有 `description` 作为路由摘要；在 API Skill `configuration` 中新增独立接口说明字段；同时新增参数格式契约字段，作为运行时校验依据。
- 原因：这样能同时满足人类编辑、Agent 路由和运行时执行三种不同消费方式，避免再把详细说明塞进 `description`。
- 备选方案：
  - 仅扩展 `description`：会继续混淆路由信息和详细说明，无法做渐进披露。
  - 只增加接口说明、不增加参数契约：仍无法满足“请求前严格校验”的需求。

### 决策 2：渐进披露在动态工具注册与调用执行之间完成

- 方案：Skill discovery 阶段仅将简要 `description` 注册给 Agent；在确定调用某个 API Skill 时，再把该 Skill 的详细接口说明拼入调用上下文或调用专用描述中。
- 原因：这样不会污染大模型的全局工具选择空间，也能在真正调用时提供足够的字段语义和字典值说明。
- 备选方案：
  - 启动时始终注册完整接口说明：会增加 prompt 噪声，降低路由质量。
  - 调用阶段完全不注入接口说明：模型仍然容易生成错误字段或错误枚举值。

### 决策 3：参数格式契约采用可序列化、可运行时校验的数据结构

- 方案：在 API Skill 配置中保存统一的参数格式定义，并在运行时转换为严格校验逻辑；前端提供结构化或 JSON 形式的维护入口，生成工具也产出同一结构。
- 原因：需要一份可持久化、可编辑、可机器校验的契约，供前端保存、后端执行和生成工具共用。
- 备选方案：
  - 只写自然语言接口说明：不具备稳定的程序化校验能力。
  - 在代码中硬编码每个 API Skill 的校验器：无法支持动态 Skill。

### 决策 4：请求前失败要返回结构化校验错误，而不是兜底发请求

- 方案：在 API Skill handler 中对模型输出先执行契约校验；失败时返回结构化错误对象，包含字段名、错误类型和预期格式，不触发外部 API 请求。
- 原因：这能把错误拦在最靠前的位置，减少无效请求和外部系统噪声，也更利于 Agent 自动修正参数。
- 备选方案：
  - 记录 warning 后继续请求：会把不确定输入传给外部接口，造成不可控结果。
  - 仅返回字符串错误：不利于前端展示和 Agent 后续重试。

### 决策 5：`skill_generator` 与手工表单使用同一目标配置协议

- 方案：`skill_generator` 输出的 API Skill 必须直接落到新的配置结构，包括路由摘要、接口说明和参数契约；缺少这些要素时返回 `INPUT_INCOMPLETE`。
- 原因：如果生成工具仍沿用旧结构，就会持续制造与手工编辑器不兼容的数据。
- 备选方案：
  - 先生成旧结构，再由后处理迁移：增加额外转换步骤和失败面。

## Risks / Trade-offs

- [配置复杂度上升] → 通过结构化表单分组、默认模板和校验提示降低录入成本。
- [旧 API Skill 与新结构并存] → 为历史配置提供兼容解析，并在无法安全映射时阻止结构化覆盖保存。
- [渐进披露实现位置不当可能导致说明丢失] → 在 spec 中明确 discovery 与 invocation 两个阶段的职责，并为 API Skill 动态工具注册补充测试。
- [参数契约设计过严导致可用性下降] → 支持默认值、可选字段和清晰错误反馈，让 Agent 能基于错误重试。
