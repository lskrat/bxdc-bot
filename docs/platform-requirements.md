## 项目概述

**bxdc-bot** 是一个 AI Agent 对话平台，让用户通过自然语言与智能助手交互，调用预定义工具完成各类自动化任务。

### 一句话描述

用户在前端输入指令 → **skill-gateway** 接收并转发至 agent-core → AI 助手理解意图 → 调用相应 Skill（API/SSH/自主规划等）→ 返回结果给用户。

**架构原则**：前端**只连接 skill-gateway**，所有请求由 skill-gateway 统一代理，agent-core 不直接对外暴露。

## 用户故事

### 故事 1：运维人员查服务器状态

> 小李是运维工程师，想查看生产环境服务器的资源使用情况。
> 
> 他在对话框输入："帮我查看生产环境服务器的 CPU 和内存使用情况"
> 
> AI 助手理解后，调用预设的 **SSH Skill**（执行固化脚本），安全地获取服务器指标并整理返回给小李。

### 故事 2：销售人员用提示词模板

> 小王需要给客户发一封产品介绍的邮件，但不知道怎么说更专业。
> 
> 他输入："帮我把这段产品特点改写成正式的客户介绍邮件：我们的产品有实时同步、多端协作、数据安全三大特点"
> 
> AI 助手调用 **提示词类 Skill**，基于预设模板生成一封格式规范、语气专业的商务邮件。

### 故事 3：产品经理查业务数据

> 小张想查看昨天的订单量。
> 
> 她输入："查询昨天创建的订单数量"
> 
> AI 助手调用 **API 类 Skill**，查询内部系统，返回统计结果。

### 故事 4：复杂任务自主规划（服务器问题排查）

> 小陈是运维工程师，收到告警说某服务响应慢。
> 
> 他输入："帮我排查 192.168.1.100 这台服务器的 Java 服务为什么响应慢"
> 
> AI 助手启动 **自主规划类 Skill**，自动拆解并执行排查步骤：
> - 调用 **"查询服务器指标"** Skill（API 类）→ 发现 CPU 正常，内存占用高
> - 调用 **"查询服务日志"** Skill（API 类）→ 发现 GC 频繁
> - 调用 **"查询应用配置"** Skill（API 类）→ 发现 JVM 堆内存配置过小
> - 汇总诊断结果，给出优化建议（调整 JVM 堆内存参数）
>
> 整个过程中，AI 根据上一步的观察结果决定下一步调用哪个 Skill，最终输出完整的排查报告。
>
> **背景**：运维团队已部署专门的运维工具服务，提供标准化 API 供 Agent 调用，无需直接 SSH 登录服务器。

### 故事 5：Skill 生成工具（扩展平台能力）

> 小刘是业务人员，经常需要查询订单系统的各种数据。每次他问 AI "帮我查一下昨天取消的订单"，AI 都回答不了，因为平台没有配置这个查询能力。
> 
> 他输入："帮我创建一个 Skill，可以查询订单系统中昨天取消的订单列表，需要调用 http://order-api.internal/api/orders/cancelled 这个接口"
> 
> AI 助手调用 **Skill 生成工具（skill_generator）**，分析小刘的需求：
> - 理解意图：查询昨日取消订单
> - 识别接口：GET /api/orders/cancelled
> - 推断参数：需要 date 参数（昨日）
> - 生成配置：构建 API Skill 配置（endpoint、method、参数映射）
> - 保存到 Skill 库：通过 `POST /api/skills` 创建新的 Extension Skill
>
> Skill 创建成功后，小刘立即测试："查询昨天取消的订单"
> 
> 这次 AI 成功调用刚创建的 Skill，返回了订单列表。
>
> **关键价值**：
> - **降低创建门槛**：用户用自然语言描述需求，AI 自动生成 Skill 配置
> - **即时可用**：创建后立即生效，无需等待开发排期
> - **持续积累**：创建的 Skill 可选择公开，供团队其他成员使用
> - **生态扩展**：业务人员自助扩展平台能力，而非依赖开发团队

### 故事 6：多对话管理（类似 DeepSeek）

> 小赵是产品经理，同时处理多个项目：
> 
> **对话 1 - 项目 A（数据分析）**：
> - 配置了 "查询订单 API"、"生成图表"、"数据导出" 等 Skill
> - 历史记录：昨天分析了用户留存率、今天正在分析付费转化漏斗
> 
> **对话 2 - 项目 B（客服支持）**：
> - 配置了 "查询客服工单"、"发送邮件"、"提示词模板" 等 Skill
> - 历史记录：上周整理了常见问题回复模板、正在处理客户投诉数据
> 
> **对话 3 - 通用问答**：
> - 只配置了基础的 Built-in Skill
> - 用于日常知识问答，不涉及具体业务系统
>
> 小赵可以随时切换对话，每个对话的：
> - **可用 Skill 集合**独立配置（避免无关 Skill 干扰）
> - **历史记录**完整保存（可往上翻看之前的对话内容）
> - **上下文**相互隔离（对话 1 的 "这个指标" 不会影响到对话 2）
>
> 当他需要新开一个分析任务时，可以选择：
> - **新建对话**：从零开始，选择需要的 Skill
> - **复制对话**：基于现有对话复制，保留 Skill 配置，清空历史
>
> **使用体验类似 DeepSeek**：左侧对话列表、右侧聊天区域、可翻看完整历史。

## 核心概念（写给开发人员）

### 1. Skill（技能）

Skill 是平台的核心能力单元。平台中的 Skill 分为两大类：

#### 1.1 分类总览

```
Skill
├── Built-in（内置）
│   └── 平台原生提供，代码实现
│       例如：api_caller、compute、ssh_executor
│
└── Extension（扩展）
    ├── 预配置类（Pre-configured）
    │   ├── API 类 - 调用外部接口（推荐）
    │   ├── SSH 类 - 执行固化脚本（受限）
    │   └── 提示词类（Template）- 基于模板生成内容
    │
    └── 自主规划类（OPENCLAW）
        └── 根据用户指令，自主调用存量 Skill 完成任务
```

#### 1.2 详细说明与示例

| 类型 | 子类型 | 说明 | 示例 |
|------|--------|------|------|
| **Built-in** | - | 平台原生提供，代码实现，所有用户可用 | `api_caller`（调用API）、`compute`（计算）、`ssh_executor`（SSH执行器） |
| **Extension** | **API 类** | 调用配置好的外部 HTTP 接口（推荐方式） | "查询服务器指标" - 调用运维平台 `/api/metrics/query` |
| **Extension** | **SSH 类** | 执行预定义、固化的 SSH 脚本，**不支持任意命令**（受限方式） | "查看服务器资源" - 执行预设的 `top` + `df` 脚本 |
| **Extension** | **提示词类** | 基于预设模板，填入用户输入后生成内容 | "邮件助手" - 将用户草稿改写成正式商务邮件 |
| **Extension** | **自主规划类** | 理解复杂需求，拆解步骤，调用存量 Skill 执行 | "服务器问题排查" - 自动调用运维 API 组合完成诊断 |

#### 1.3 Skill 可见性（公开/私有）

- **私有 Skill**：创建者自己使用
- **公开 Skill**：创建者选择公开后，其他用户也能在他们的对话中使用

**功能边界**：
- Built-in 是平台能力的基础，相对稳定，所有用户默认可用
- Extension 是平台能力的延伸，用户可自主创建，可选择公开分享
- 对 AI 模型来说，Built-in 和 Extension 没有区别，都是可调用的工具

### 2. SSH 安全模型（重要）

为保障系统安全，平台对 SSH 操作有严格限制：

| 能力 | 支持 | 说明 |
|------|------|------|
| 固化脚本执行 | ✅ | Skill 内预定义脚本，如查资源、看日志 |
| 任意命令执行 | ❌ | 不支持用户输入 `rm -rf` 等任意命令 |

**实现方式**：
- SSH Skill 在配置时固化脚本内容（如 `df -h`、`ps aux`）
- 执行时只传服务器标识，不传命令内容
- 命令白名单在 skill-gateway 端控制

### 3. ReAct（推理-行动循环）

这是 AI 助手的工作模式：

```
用户提问 → 模型思考（Thought）→ 决定行动（Action）→ 执行工具 → 观察结果（Observation）→ 继续思考...
```

**功能边界**：
- agent-core 负责驱动这个循环
- 每轮循环可能调用 0 个或多个工具
- 循环直到模型认为可以给出最终答案为止
- **不支持代码编写**：工具只能调用预定义 Skill，不能生成并执行代码

### 4. 自主规划类 Skill（OPENCLAW）

这是 Extension 中的一种特殊类型：

**特点**：
- 接收复杂、多步骤的用户指令
- AI 自主拆解任务，决定调用哪些存量 Skill（可以是 Built-in 或其他 Extension）
- 按顺序或依赖关系执行，汇总结果

**示例流程（服务器问题排查）**：
```
用户："帮我排查 192.168.1.100 的 Java 服务为什么响应慢"

OPENCLAW Skill 自主规划执行：
1. 调用 "查询服务器指标" Skill（API 类）
   → POST 运维平台 /api/metrics/query
   → 观察：CPU 正常，内存占用 95%
2. 调用 "查询服务日志" Skill（API 类）
   → POST 日志平台 /api/logs/search
   → 观察：发现 Full GC 每分钟 20 次
3. 调用 "查询应用配置" Skill（API 类）
   → POST 配置中心 /api/apps/config
   → 观察：JVM 堆内存只配置了 512m
4. 综合诊断：堆内存不足导致 GC 频繁，建议调整为 -Xmx2g
5. 输出完整排查报告
```

**关键特点**：
- 每步执行后，AI 根据观察结果**自主决定**下一步调用哪个 Skill
- 如果第 1 步发现 CPU 100%，可能会改调用 "查询热点进程" Skill 而不是 "查日志"
- 所有调用的 Skill 必须是用户有权限的**存量预配置 API Skill**
- 实际执行由运维平台、日志平台等外部服务完成，Agent 只通过标准化 API 获取结果

**功能边界**：
- 只能调用**已存在**的 Skill（Built-in 或 Extension）
- 不能创建新 Skill 或编写代码
- 规划范围限于当前用户有权限使用的 Skill

### 5. 记忆（Memory）

平台通过 mem0 服务存储用户相关信息：

- **长期记忆**：用户的偏好、常用服务器、历史上下文等
- **短期记忆**：当前对话的最近几轮内容

**功能边界**：
- 记忆服务是独立的，agent-core 按需调用
- 用户可以说 "帮我查查上次那个问题"，平台能回忆起来
- 敏感信息（密码等）**不应该**存入记忆

### 6. 用户日志记录（Audit Log）

平台在数据库中记录用户交互全过程，用于审计、追踪和问题排查：

| 记录类型 | 内容 | 存储位置 |
|----------|------|----------|
| **对话记录** | 用户输入、AI 回复、时间戳、会话 ID | 数据库 |
| **工具调用记录** | 调用的 Skill 名称、输入参数、返回结果、调用耗时、是否成功 | 数据库 |
| **LLM 交互记录** | 请求/响应内容（可配置脱敏和截断策略）| 数据库或本地日志 |

**功能边界**：
- 记录由 **skill-gateway** 统一写入数据库（agent-core 通过 HTTP 上报，不直连数据库）
- 支持按用户、时间、Skill 类型等多维度查询
- 敏感参数（如密码、Token）可配置脱敏规则
- 日志保留策略可配置（如保留 90 天自动归档）

**典型使用场景**：
- 用户反馈 "AI 昨天给了一个错误的建议" → 通过日志快速定位对话内容和调用链
- 安全审计 → 查看某个敏感 Skill 被谁、在什么时间、用什么参数调用过

### 7. 多对话管理（Conversation Management）

用户可以创建并管理多个独立的对话会话，每个对话拥有自己的配置和历史：

**核心能力**：

| 能力 | 说明 | 示例 |
|------|------|------|
| **多会话并行** | 用户可同时维护多个活跃对话 | 同时有 "项目 A 分析"、"项目 B 运维"、"日常问答" 三个对话 |
| **独立 Skill 配置** | 每个对话可配置不同的可用 Skill 集合 | 对话 1 只启用数据分析类 Skill，对话 2 只启用运维类 Skill |
| **完整历史保存** | 对话的每一轮交互都持久化存储 | 可随时往上翻看之前的问答内容 |
| **上下文隔离** | 各对话之间的上下文互不干扰 | 对话 1 中的 "这个问题" 不会影响到对话 2 |
| **会话生命周期** | 支持新建、切换、重命名、删除对话 | 类似 DeepSeek 的左侧对话列表管理 |

**对话配置（Conversation Config）**：

每个对话可独立配置：
```yaml
conversation:
  name: "项目A数据分析"           # 对话名称
  description: "分析Q3用户留存"    # 可选描述
  enabled_skills:                 # 该对话可用的 Skill 列表
    - "查询订单API"               # Extension Skill
    - "生成图表"                 # Extension Skill
    - "compute"                  # Built-in Skill
    - "api_caller"               # Built-in Skill
  memory_enabled: true            # 是否启用长期记忆
  confirmation_mode: "default"    # 确认模式：default / strict / relaxed
```

**使用场景**：

| 场景 | 说明 |
|------|------|
| **按项目隔离** | 不同项目使用不同的 Skill 集合和历史上下文 |
| **按角色区分** | 作为开发人员时启用运维 Skill，作为业务人员时启用数据查询 Skill |
| **按任务分离** | 一个对话专注做数据分析，另一个专注写文档，互不干扰 |
| **历史追溯** | 两周前分析过的数据，通过历史记录快速回顾当时的结论 |

**与记忆的关系**：
- **对话历史**：完整的多轮对话内容（短期记忆的作用域）
- **长期记忆（mem0）**：跨对话共享的用户偏好和关键信息
- 示例：用户在对话 1 提到 "我的主力服务器是 srv-001"，对话 2 也能通过记忆知道这一点

**功能边界**：
- 每个用户可创建的对话数量可配置上限（如最多 50 个）
- 单条对话的历史轮数可配置上限（如最多保留 1000 轮）
- Skill 配置在对话创建时初始化，对话进行中可动态调整可用 Skill
- 删除对话会级联删除该对话的所有历史记录（但审计日志保留）

### 9. Skill 运行时确认（Confirmation）

对于高风险的 Skill（如涉及删除、修改、重要数据变更），支持配置**运行时确认**机制：

**工作流程**：
```
1. 用户输入指令 → AI 决定调用某 Skill
2. 检查 Skill 配置：requiresConfirmation = true？
3. 如果是：暂停执行，向前端发送确认请求
   - 展示内容：Skill 名称、作用说明、本次调用的参数值
   - 示例："即将执行 '删除服务器日志'，目标服务器：192.168.1.100，日志路径：/var/log/app，是否确认？"
4. 用户点击「确认」→ 继续执行
5. 用户点击「取消」→ 终止执行，AI 回复 "操作已取消"
```

**功能边界**：
- 确认配置在 **Skill 级别**（每个 Skill 独立配置是否需要确认）
- 确认粒度到**单次调用**（不是整个对话）
- 前端需支持中断/恢复机制（SSE 流暂停等待用户输入）
- 超时机制：用户 N 秒内未确认，自动取消

**典型应用场景**：
| Skill 类型 | 是否需要确认 | 说明 |
|------------|--------------|------|
| 查询类（查日志、查指标）| 否 | 只读操作，无风险 |
| 删除类（删日志、清缓存）| 是 | 破坏性操作，需二次确认 |
| 修改类（改配置、重启服务）| 是 | 影响系统状态，需确认 |
| 发送类（发邮件、发短信）| 可选 | 根据内容敏感度决定 |

|------|------|------|----------|
| **接入层** | **前端** | 用户界面、展示对话 | 面向用户 |
| **网关层** | **skill-gateway** | 统一入口、会话管理、请求代理、Skill 管理、执行调度 | **唯一对外服务** |
| **推理层** | **agent-core** | ReAct 循环、LLM 交互、工具编排、记忆检索 | **仅内部调用，不直接对外** |
| **数据层** | **mem0 / MySQL** | 记忆存储、对话历史、审计日志 | 内部服务 |
| **执行层** | **外部服务** | SSH、运维平台 API、业务系统 API | 通过 skill-gateway 代理访问 |

**关键设计要点**：

1. **统一入口**：skill-gateway 是系统的**唯一对外入口**，所有流量（用户请求、内部调用）都必须经过它
2. **安全隔离**：agent-core 不直接对外暴露，避免 LLM 推理层直接承受公网流量
3. **会话管理**：对话创建、历史查询、配置管理等由 skill-gateway 负责，agent-core 专注推理
4. **请求代理**：skill-gateway 接收用户请求 → 转发至 agent-core → 接收推理结果 → 返回给前端
5. **日志统一**：所有审计日志由 skill-gateway 统一写入，便于集中管理和查询

## 功能边界（能做什么 / 不能做什么）

### ✅ 平台能做的

1. **自然语言理解**：理解用户意图，拆解成可执行步骤
2. **预定义工具调用**：调用配置好的 Skill（API、SSH 固化脚本、提示词模板等）
3. **多轮对话**：根据上下文持续交互，直到任务完成
4. **记忆利用**：记住用户偏好和历史，提供个性化体验
5. **自主规划**：复杂任务拆解，自动调用存量 Skill 组合完成
6. **服务器操作**：
   - 通过 **API 类 Skill** 调用运维平台等外部服务（推荐方式）
   - 通过 **固化的 SSH Skill** 执行预定义脚本（受限方式）
7. **Skill 共享**：用户可创建 Skill 并选择公开，供他人使用
8. **用户日志记录**：在数据库中记录对话历史、工具调用详情，支持审计和问题排查
9. **多对话管理**：用户可以创建多个对话，每个对话独立配置可用 Skill、保存完整历史
10. **Skill 生成**：用户通过自然语言描述，AI 自动生成新的 Extension Skill
11. **运行时确认**：高风险 Skill 支持配置二次确认，防止误操作

### ❌ 平台不做/不擅长的

1. **代码编写与执行**：不支持生成并运行代码，所有操作必须通过预定义 Skill
2. **任意 SSH 命令**：为保障安全，SSH 只能执行固化在 Skill 内的脚本，不支持自由输入命令
3. **直接数据库操作**：agent-core 不直连业务库，只能通过 API/SSH 间接操作
4. **长时任务保持**：对话结束后的后台任务需外部调度支持
5. **文件系统直接操作**：文件操作需通过特定工具，不能直接访问任意路径
6. **实时流式大数据处理**：适合小批量数据，不适合大规模实时计算
7. **用户权限管理细粒度控制**：基础鉴鉴权有，复杂 RBAC 需扩展

## 数据流向（典型场景）

### 场景：用户让 AI 排查服务器问题（自主规划类 Skill）

```
1. 用户输入："帮我排查 192.168.1.100 的 Java 服务为什么响应慢"
   → 前端 → skill-gateway（统一入口接收）

2. skill-gateway 检查会话有效性，记录请求日志
   → 转发至 agent-core

3. agent-core 检索记忆（可选）→ mem0

4. 模型启动自主规划类 Skill（OPENCLAW），开始多步排查：

   第 1 轮：
   - agent-core 决定调用 "查询服务器指标" API Skill
   - 调用 → skill-gateway（执行层）
   - skill-gateway 转发至运维平台 /api/metrics/query
   - 返回：CPU 正常，内存 95%
   - AI 观察：内存占用高，需要查具体原因

   第 2 轮：
   - agent-core 决定调用 "查询服务日志" API Skill
   - 调用 → skill-gateway
   - skill-gateway 转发至日志平台 /api/logs/search
   - 返回：Full GC 频繁
   - AI 观察：怀疑 JVM 内存配置不足

   第 3 轮：
   - agent-core 决定调用 "查询应用配置" API Skill
   - 调用 → skill-gateway
   - skill-gateway 转发至配置中心 /api/apps/config
   - 返回：-Xmx512m
   - AI 观察：确认堆内存配置过小

5. 自主规划 Skill 汇总观察结果：
   "诊断结论：JVM 堆内存配置（512MB）不足导致频繁 Full GC，建议调整为 2GB"

6. 最终结果 → agent-core → skill-gateway → 前端 → 用户看到排查报告

7. skill-gateway 记录完整对话历史和工具调用记录 → 数据库
8. agent-core 记忆本次交互 → mem0
```

**数据流向要点**：
| 步骤 | 流向 | 说明 |
|------|------|------|
| 用户请求 | 前端 → **skill-gateway** → agent-core | 前端**不直连** agent-core |
| Skill 调用 | agent-core → **skill-gateway** → 外部服务 | agent-core 不直接调用外部，必须通过网关 |
| 日志记录 | agent-core 上报 → **skill-gateway** → 数据库 | 统一由 skill-gateway 落库 |
| 响应返回 | agent-core → **skill-gateway** → 前端 | 统一出口 |

**注意**：
- 所有进出 agent-core 的流量都经过 skill-gateway，实现统一管控
- skill-gateway 负责鉴权、限流、日志、审计等横切面功能
- agent-core 专注 LLM 推理和 ReAct 循环，不感知公网流量

## 给开发人员的要点

### 新架构核心：skill-gateway 作为统一入口

**所有流量必须经过 skill-gateway**：

```
前端 ──HTTP/SSE──► skill-gateway ──HTTP──► agent-core
                       │
                       ├─ 鉴权、限流
                       ├─ 会话管理
                       ├─ 请求/响应代理
                       ├─ 统一日志落库
                       └─ Skill 元数据管理
```

**关键实现要点**：

| 功能 | skill-gateway 职责 | agent-core 职责 |
|------|-------------------|-----------------|
| **对外接口** | 提供 `/api/agent/run` 等统一入口 | **不对外暴露** |
| **会话管理** | 创建/查询对话、管理会话状态 | 接收 `conversationId` 参数 |
| **请求代理** | 接收用户消息，转发至 agent-core | 处理推理，返回结果 |
| **响应流** | 接收 agent-core SSE，转发给前端 | 生成 SSE 事件流 |
| **日志记录** | 统一写入数据库 | 通过 HTTP 上报 |
| **Skill 执行** | 接收 agent-core 调用，代理至外部服务 | 发起调用请求 |

**实现示例**：

1. **前端请求路径变化**：
   ```javascript
   // 旧：直连 agent-core
   fetch('http://agent-core:3000/agent/run', {...})
   
   // 新：经过 skill-gateway
   fetch('http://skill-gateway:8080/api/agent/run', {...})
   ```

2. **skill-gateway 代理层实现**：
   ```java
   @PostMapping("/api/agent/run")
   public SseEmitter agentRun(@RequestBody AgentRequest req) {
       // 1. 鉴权、限流
       // 2. 记录请求日志
       // 3. 转发至 agent-core
       SseEmitter emitter = new SseEmitter();
       agentCoreClient.streamRun(req, event -> {
           // 4. 接收 agent-core SSE
           // 5. 记录响应日志
           emitter.send(event);
       });
       return emitter;
   }
   ```

3. **agent-core 配置变化**：
   ```typescript
   // 只监听内部网络，不对外暴露端口
   // 或：只接受来自 skill-gateway 的请求
   ```

### 如果要加新 Skill

| Skill 类型 | 修改位置 |
|------------|----------|
| **Built-in** | agent-core 的 baseTools + skill-gateway 的执行逻辑 |
| **Extension - API 类** | 在 skill-gateway 配置接口参数（URL、method、headers、参数映射），agent-core 自动加载 |
| **Extension - SSH 类** | 在 skill-gateway 配置服务器 + 固化脚本（受限场景使用） |
| **Extension - 提示词类** | 在 skill-gateway 配置 prompt 模板 |
| **Extension - 自主规划类** | 配置 orchestration 模式（serial 等），指定可调用的存量 Skill 白名单 |

### 运维场景 Skill 设计建议

对于服务器排查、监控、日志分析等运维场景：
- **优先使用 API 类 Skill**：对接现有运维平台（Prometheus、ELK、CMDB 等）
- 配置示例：
  ```yaml
  name: "查询服务器指标"
  kind: api
  endpoint: "http://ops-platform.internal/api/metrics/query"
  method: POST
  headers:
    Authorization: "Bearer ${OPS_TOKEN}"
  ```
- **SSH 类仅作为兜底**：当某些老旧系统无 API 时，使用固化脚本访问

### SSH 安全开发注意

- SSH Skill **必须**在配置阶段固化命令，执行阶段只传服务器标识
- 危险命令（如删除、修改类）需要实现确认流（interrupt/confirm）
- 参考：`backend/agent-core/src/tools/java-skills.ts` 中的 `isDangerousCommand` 逻辑

### 如果要改 ReAct 逻辑

- 入口在 `agent-core/src/agent/agent.ts`
- 工具注册在 `AgentFactory.createAgent`
- 执行分支在 `java-skills.ts`（各工具的实现）

### Skill 可见性实现

- 公开/私有字段在 `skills` 表（或其他扩展表）
- 加载时根据用户权限过滤：自己的 + 公开的 + 系统内置
- 参考：`loadGatewayExtendedTools` 中的过滤逻辑

### 自主规划类 Skill 实现

- 配置 `executionMode: OPENCLAW` 或 `kind: openclaw`
- 配置 `allowedTools` 白名单（限制可调用的存量 Skill）
- 实现参考：`executeOpenClawSkill` 在 `java-skills.ts`

### 如果要改记忆

- 记忆接口：`backend/agent-core/src/mem/memory.service.ts`
- mem0 服务独立部署，通过 HTTP 调用

### 如果要实现用户日志记录

- **写入路径**：agent-core 通过 HTTP 上报 → skill-gateway → 数据库
- **关键表**：`conversation_logs`（对话）、`tool_invocation_logs`（工具调用）
- **实现参考**：`openspec/changes/archive/2026-04-15-llm-org-log-gateway-persistence/`
- **注意**：LLM 原始 HTTP 日志量大，需配置截断和脱敏策略

### 如果要实现 Skill 运行时确认

- **配置层**：在 Skill 配置中增加 `requiresConfirmation: boolean` 字段
- **Agent 层**：在 `executeConfiguredApiSkill` / `executeServerResourceStatusSkill` 等入口检查确认标志
- **实现机制**：使用 LangGraph `interrupt` 机制暂停流程，等待前端确认
- **前端交互**：SSE 流暂停 → 展示确认弹窗 → 用户点击 → 恢复流程
- **参考实现**：`applyExtendedSkillConfirmationGate` 在 `java-skills.ts`

### 如果要实现多对话管理

- **数据模型**：
  - `conversations` 表：会话 ID、用户 ID、名称、描述、创建时间、更新时间
  - `conversation_skills` 表：会话与 Skill 的多对多关系
  - `conversation_history` 表：每轮对话内容（用户输入、AI 回复、工具调用）
  
- **API 设计**：
  - `GET /api/conversations` - 列出用户的所有对话
  - `POST /api/conversations` - 创建新对话
  - `GET /api/conversations/:id` - 获取对话详情和历史
  - `PUT /api/conversations/:id/skills` - 配置对话可用的 Skill
  - `DELETE /api/conversations/:id` - 删除对话

- **Agent 层修改**：
  - `AgentFactory.createAgent` 接受 `conversationId` 参数
  - 根据对话配置过滤可用 Skill（`enabled_skills` 列表）
  - 对话历史注入到 ReAct 循环的上下文

- **前端实现**：
  - 左侧对话列表（类似 DeepSeek）
  - 对话设置面板：配置可用 Skill、重命名、删除
  - 历史消息支持往上翻看（分页加载）

### 如果要实现 Skill 生成工具

- **核心组件**：`skill_generator`（Built-in Skill）
- **实现位置**：`backend/agent-core/src/tools/java-skills.ts` 中的 `JavaSkillGeneratorTool`

- **工作流程**：
  1. 用户用自然语言描述想要的 Skill
  2. LLM 分析需求，提取关键信息：
     - 意图（查询/操作/生成）
     - 接口信息（URL、method）
     - 参数结构（输入字段、类型）
  3. 生成 Extension Skill 配置（JSON 格式）
  4. 调用 `POST /api/skills` 保存到 Skill 库
  5. 返回创建结果给用户

- **关键能力**：
  - **意图识别**：区分 API 类、提示词类、自主规划类
  - **接口解析**：从自然语言中提取 endpoint、method、headers
  - **参数推断**：根据接口描述推断参数名称、类型、是否必填
  - **示例生成**：自动生成参数示例和返回值示例

- **安全边界**：
  - 生成的 Skill 默认私有，仅创建者可用
  - 危险操作（DELETE、敏感数据）需标记 `requiresConfirmation: true`
  - 接口域名白名单限制（防止生成恶意 Skill）

- **与扩展体系的关系**：
  - `skill_generator` 是 Built-in，**创建** Skill
  - 创建的是 Extension，**被**后续对话调用
  - 形成闭环：用户描述 → 生成 Skill → 立即使用 → 可选公开
